# Spec: siakam-attackpath-analysis Skill

**Date**: 2026-05-19
**Status**: design-approved (revised after 10-round review)

## Overview

A Claude Code skill for automated security analysis of C codebases (Linux kernel drivers, boot chain firmware, mobile co-processor firmware). The skill identifies attack paths from externally-exposed entry functions, discovers security vulnerabilities along those paths, and eliminates false positives through a three-layer review protocol.

The skill is composed entirely of markdown documents (prompts) and a single Python helper script (`cg_helper.py`). All analysis is performed by the LLM following the skill's instructions.

## Terminology Glossary

Defined in SKILL.md and referenced by all step documents.

| Term | Definition |
|------|------------|
| **Entry (entry function)** | An externally-exposed function listed in `apis.json` that can receive attacker-controlled data |
| **Infected function** | A function whose parameters can be traced back to entry's external input (data flow) |
| **Attack path** | A root-to-leaf chain of infected functions in the pruned call graph |
| **Pruned attack graph** | The call graph filtered to only infected nodes and their edges |
| **Leaf node** | An infected function with no infected callees; data flow terminus |
| **Source (label)** | The entry point itself; origin of attacker data |
| **Active (label)** | A function that reads, parses, or decides based on external input |
| **Pass-through (label)** | A function that forwards external input to callees without reading it |
| **Finding** | A potential vulnerability discovered in Step 2, pending Step 3 review |
| **CONFIRMED** | Finding verified as a real vulnerability after Step 3 review |
| **FALSE_POSITIVE** | Finding determined to be non-exploitable after Step 3 review |

## Skill File Structure

```
siakam-attackpath-analysis/
  SKILL.md                     # Entry point: arg parsing, env checks, phase orchestration, glossary
  steps/
    step1_attack_path.md       # Attack path identification (data-flow tracing + call graph pruning)
    step2_vuln_analysis.md     # Vulnerability discovery (per-function review against category system)
    step3_false_positive.md    # False-positive elimination (3-layer protocol) + final consolidation
  tools/
    cg_helper.py               # Call-graph query tool (caller/callee lookup)
```

SKILL.md is written in a **programmatic, linear-instruction style**: "Step 1: do X. Step 2: do Y." The LLM follows instructions strictly with minimal autonomous deviation.

Each step document opens with an abbreviated glossary repeating the terms relevant to that step.

## Configurable Parameters

Defined prominently at the top of SKILL.md and step1_attack_path.md for easy modification:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `MAX_CALL_DEPTH` | 10 | Maximum depth of attack path (entry → ... → leaf) |
| `MAX_INFECTED_FUNCTIONS` | 50 | Maximum infected functions per entry before truncation |

## Usage

```
/siakam-attackpath-analysis <project_dir>
```

- `project_dir` defaults to current directory if omitted
- Reads `.siakamignore` (same syntax as `.gitignore`) to exclude files/directories
- Requires `apis.json` (see Input Files) — aborts if missing
- Optionally uses `callgraph.json` if present (see Input Files)

## Input Files

### apis.json

**Default path**: `<project_dir>/.siakam_out/SII/apis.json`

Provides the list of entry functions (external attack surface). If the file does not exist at the default path, prompt the user to specify the path. If the `interfaces` array is empty, warn and exit with: "No entry functions found in apis.json. Check SII analysis results."

The `failures`, `errors`, and `warnings` fields in apis.json are ignored.

Format:

```json
{
  "project": "<basename>",
  "analysis_date": "<YYYY-MM-DD>",
  "total_candidates": <N>,
  "confirmed_interfaces": <X>,
  "summary": {
    "high_confidence": <H>,
    "medium_confidence": <M>,
    "exclusion_reasons": {}
  },
  "interfaces": [
    {
      "name": "<function name>",
      "file": "<path relative to project_dir>",
      "line": <line>,
      "confidence": "high | medium",
      "analysis": "<multi-line analysis>"
    }
  ],
  "failures": [],
  "errors": [],
  "warnings": []
}
```

### callgraph.json

**Default path**: `<project_dir>/.siakam_out/callgraph.json`

Optional pre-built call graph generated by external Siakam framework modules (e.g., a call-graph construction tool). The skill does NOT generate callgraph.json — it only consumes it if present. If absent, the skill builds the call graph from source code during Step 1.

Nodes represent functions with location metadata; edges can be `direct` or `indirect` (indirect edges include `uid` and `confidence`). See the sample `callgraph.json` in the skill root for the full schema.

### .siakamignore

**Path**: `<project_dir>/.siakamignore`

Same syntax as `.gitignore`. Excluded files/directories are skipped entirely — they are not read, not analyzed, and any functions from excluded files that appear in callgraph.json edges are also excluded from analysis.

## cg_helper.py Contract

### Interface

```
python3 cg_helper.py <FUNC> caller   # List all functions that call FUNC
python3 cg_helper.py <FUNC> callee   # List all functions called by FUNC
```

### Input

- `FUNC`: A function name string (exact match)
- The tool reads `callgraph.json` from the current working directory

### Output (stdout)

JSON array of matching edges:

```json
[
  {
    "caller": "<caller_func>",
    "callee": "<callee_func>",
    "type": "direct | indirect",
    "file": "<source_file>",
    "line": <line_number>,
    "uid": "<unique_id>",
    "confidence": "high | medium | low"
  }
]
```

- `uid` and `confidence` are present only for indirect edges
- Returns empty array `[]` if no matches found
- Returns error JSON `{"error": "<message>"}` if callgraph.json is missing or malformed

## Step 1: Attack Path Identification

**Goal**: For each entry function, construct a pruned call graph containing only functions affected by attacker-controlled data. Each root-to-leaf path in the pruned graph is an attack path.

### Key Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `MAX_CALL_DEPTH` | 10 | Maximum nodes in an attack path (entry → ... → leaf) |
| `MAX_INFECTED_FUNCTIONS` | 50 | Maximum infected functions per entry; analysis truncated if exceeded |

### Call Graph Construction Strategy

**When callgraph.json is available:**
1. Load the full call graph via `cg_helper.py` queries (BFS from entry)
2. Build the complete call graph up to `MAX_CALL_DEPTH` levels in one pass
3. Then analyze data flow for each node to determine infection status
4. Verify indirect edges: `high` confidence edges are accepted directly; `medium` and `low` confidence edges receive auxiliary semantic analysis but the path is kept even if uncertain (confidence annotation carried forward)

**When callgraph.json is NOT available:**
1. Start from entry function as root
2. Iterative BFS layer by layer: for each function at the current level, read its source body, identify all callees (function calls, function pointer assignments), determine data flow infection status, write intermediate results to `.siakam_out/SAA/.step1_state/<uid>_layer_<N>.json`
3. Only continue expanding from infected functions
4. Each layer's state file is read by the next layer iteration

**Both modes** use task-list + iteration: `.siakam_out/SAA/tasks.md` tracks each function's data-flow analysis status. No function is forgotten between iterations.

### Call Graph Construction Rules

1. Start from entry function as root
2. Standard C library and kernel API functions are terminal (leaf) nodes: `malloc`, `free`, `kmalloc`, `kfree`, `copy_from_user`, `copy_to_user`, `printk`, `printf`, `sprintf`, `memcpy`, `memmove`, `memset`, `strcpy`, `strncpy`, `strlen`, and similar. Do not expand past these.
3. **Cycle detection**: Track visited functions per path. When a function is encountered that already appears earlier in the same path (back edge), terminate that branch and mark it `[terminated at cycle: <func>]`.
4. **Header file entries**: If an entry function is defined in a `.h` file, it is analyzed normally but macro-wrapped logic is not deeply expanded.

### Data Flow Tracing Rules

**Fundamental principle**: "Reception = Infection" — if a function's parameter can be traced back to the entry's external input, the function is infected, regardless of whether it actively uses the data.

**Tracing rules** (in priority order):

1. **Direct pass**: Parameter passed directly to child function as argument → child infected
2. **Local assignment**: Parameter assigned to local variable, local variable passed to child → child infected
3. **Struct field propagation**: Parameter written to struct field, struct passed to child that reads the field → child infected
4. **Pointer alias propagation**: If pointer P points to attacker data and `Q = P ± offset` (pointer arithmetic), Q passed to child → child infected. Same for type-cast aliases (`void* → struct*`).
5. **Memory copy propagation**: Parameter data copied via memcpy/strcpy/custom copy functions, the destination buffer passed to child → child infected
6. **Control-flow influence**: Parameter used in branch condition, child called within branch → child infected (indirect influence)
7. **Comparison-only exclusion**: Parameter used only in comparison where result does not affect child function invocation → child NOT infected

**Termination**: Data flow terminates when the external-input-derived data no longer participates in any function-call argument computation.

### Custom Data-Transfer Function Recognition

Before tracing, LLM must scan source code for project-specific data-transfer functions that serve as data-flow conduits:

- Custom memory copy functions (name patterns: copy/mem/move/transfer; body contains memcpy/memmove/strcpy)
- Hardware data-read functions (DMA buffer, FIFO, MMIO reads)
- Serialization/deserialization functions (message pack/unpack, payload marshalling)
- IPC/shared-memory transfer functions (cross-core communication, shared memory writes)

These functions propagate data just like standard memcpy — if F transfers data from location X to Y, and G can read from Y, then G is infected.

### Pruned Attack Graph

- **Nodes**: Only infected functions
- **Edges**: Only call relationships between infected functions
- **Attack path** = root (entry) → leaf (any infected node with out-degree 0 in pruned graph)
- **Empty result**: If no infected functions beyond the entry itself are found, the entry is still a valid attack path of length 1 (the entry function itself is both root and leaf). Generate the standard output with 1 path, 1 node. The entry function's own code is still analyzed for vulnerabilities in Step 2.

### Node Classification Labels

| Label | Meaning |
|-------|---------|
| source | Entry point, origin of attacker data |
| active | Actively reads, parses, or makes decisions based on external input |
| pass-through | Forwards external input to children without reading it |
| leaf | Data flow terminus (no further infected callees) |

### Truncation

If the number of infected functions exceeds `MAX_INFECTED_FUNCTIONS`, stop expansion and annotate the report with:

```
**WARNING**: Analysis truncated at <N> infected functions (limit: MAX_INFECTED_FUNCTIONS).
<M> functions were not analyzed.
```

## Step 2: Vulnerability Discovery

**Goal**: For each attack path, review every function along the path for security vulnerabilities against a structured category system.

### Execution Model

- If an entry has more than 10 attack paths, group paths by infected function (not by path count) to avoid multi-agent re-analysis of the same function. Each sub-agent handles one function group.
- Otherwise: one sub-agent per entry (all attack paths for that entry analyzed together)
- Collect all functions across the assigned paths → deduplicate (same name + same file = same function)
- Sort by call-graph depth (shallow first), analyze each function once
- Read the `## Function Index` from the attack_path.md to locate all functions efficiently

### Per-Function Review Protocol

For each function, read its full source body. First, scan the function body to determine which analysis methods are applicable. Then, for each applicable method, perform the analysis and record the result. Every method must be explicitly marked.

**Checklist output per function** (8 methods, each marked):

| Method | Applicable? | Result | Notes |
|--------|-------------|--------|-------|
| Source-to-sink tracking | YES/NO | PASS/FAIL/N/A | ... |
| Bounds-check completeness | YES/NO | PASS/FAIL/N/A | ... |
| Integer safety | YES/NO | PASS/FAIL/N/A | ... |
| Lifetime analysis | YES/NO | PASS/FAIL/N/A | ... |
| Synchronization check | YES/NO | PASS/FAIL/N/A | ... |
| Authorization gating | YES/NO | PASS/FAIL/N/A | ... |
| Secret pattern matching | YES/NO | PASS/FAIL/N/A | ... |
| Cross-trust-boundary checks | YES/NO | PASS/FAIL/N/A | ... |

**Method details:**

| Method | Description | Target Categories |
|--------|-------------|-------------------|
| Source-to-sink tracking | Trace entry parameter (taint source) to dangerous operation sinks (memcpy, sprintf, copy_to_user, mmio_write, etc.) | Input validation, memory safety |
| Bounds-check completeness | For memcpy(dst, src, len) / array[idx], trace backward to verify len/idx underwent bounds validation before use | Memory safety |
| Integer safety | Check arithmetic on tainted data (addition, multiplication, offset calculation) for overflow/underflow, especially before memory operations | Memory safety |
| Lifetime analysis | Check malloc/kfree pairing, whether freed pointers are nullified, whether freed pointers are reused | UAF, double-free |
| Synchronization check | Verify shared-variable access in multi-threaded contexts is lock-protected | Concurrency |
| Authorization gating | Check whether security-sensitive operations (physical memory access, register writes, DMA mapping) are guarded by capability/permission checks | System security, firmware |
| Secret pattern matching | Search for hardcoded keys, certificate fingerprints, fixed IVs/nonce values | Cryptographic |
| Cross-trust-boundary checks | Verify shared-memory / cross-trust-domain data has integrity checks (CRC, signature, magic number) before use | Input validation, firmware |

### Vulnerability Category System

**A. Input Validation**
- Missing or insufficient validation on interface parameters
- Shared-memory data read without integrity/safety checks
- Trust boundary violations (trusting data from untrusted sources)
- Missing bounds checking on externally-sourced data
- Type confusion from unvalidated input
- Deserialization of untrusted data without validation
- Missing validation on IPC data
- Environment variable injection from external sources

**B. Memory Safety**
- Stack-based buffer overflow
- Heap-based buffer overflow
- Out-of-bounds read/write (array index vulnerabilities)
- Use-after-free (UAF)
- Double free
- Null pointer dereference
- Uninitialized memory access
- Integer overflow/underflow leading to memory corruption
- Format string vulnerabilities

**C. System Security**
- Authentication bypass logic
- Privilege escalation paths
- Remote code execution
- Dynamic code execution

**D. Cryptographic Security**
- Hardcoded API keys, passwords, or tokens
- Weak cryptographic algorithms or implementations
- Improper key storage or management
- Cryptographic randomness issues
- Certificate validation bypasses

**E. Firmware/Embedded-Specific**
- DMA attack surface (unvalidated DMA buffers)
- Improper MMIO access control
- Physical memory mapping security
- Secure boot bypass
- Improper hardware register exposure
- TrustZone/PMP boundary violations

**F. Concurrency (Multi-Threaded)**
- TOCTOU with clear exploitable window
- Shared state accessed without synchronization
- Demonstrable data race with security impact

### Finding Consolidation Rule

When multiple vulnerability types share the same root cause (e.g., integer overflow leading to buffer overflow), report them as a **single composite finding**. The Description must explain the causal chain. Do not report each downstream consequence as a separate finding.

### Scoring

**Severity:**
- HIGH: Directly exploitable — leads to RCE, privilege escalation, or authentication bypass
- MEDIUM: Requires specific conditions but has significant impact
- LOW: Defense-in-depth issues or lower-impact vulnerabilities

**Confidence:**
- 0.9–1.0: Certain exploit path, potentially tested
- 0.8–0.9: Clear vulnerability pattern with known exploitation methods
- 0.7–0.8: Suspicious pattern requiring specific conditions
- Below 0.7: Do NOT report

## Step 3: False-Positive Elimination

**Goal**: Rigorously re-examine every finding with fresh context. Three-layer protocol.

**Key principle**: "Better to miss some theoretical issues than flood the report with false positives."

**Operational rule**: Do NOT run commands to reproduce findings. Do NOT use Bash tool. Do NOT write files directly (sub-agents return review conclusions; the main agent updates the `<uid>_vuls.md` files).

**Context isolation**: Each step document includes the hard constraint:
> "Use ONLY the input files specified in this section. Do not rely on conclusions or judgments from previous analysis phases. Re-examine from first principles."

### Layer 1: Hard Exclusion Rules

Automatically exclude findings matching these patterns (do not pass to Layer 2):

- Denial of Service (DOS) or resource exhaustion
- Secrets/credentials stored on disk (if disk is otherwise secured)
- Rate limiting concerns or service overload scenarios
- Memory consumption or CPU exhaustion
- Lack of input validation on non-security-critical fields without proven security impact
- Lack of hardening measures (not a concrete vulnerability)
- Purely theoretical race conditions without concrete exploitation path
- Vulnerabilities in outdated third-party libraries (managed separately)
- Files that are unit tests or test-only code
- Log spoofing (un-sanitized user input in logs)
- User-controlled content in AI system prompts
- Regex injection
- Insecure documentation (markdown files, etc.)
- Lack of audit logs

### Layer 2: Context Re-Verification

For each finding that passes Layer 1, re-read source code with a clean perspective (do not rely on Step 1/2 conclusions). Verify:

| Check | Method |
|-------|--------|
| Reachability | Is the call path real? Are indirect calls confirmed? Does the function pointer actually resolve to this function? |
| Data-flow validity | Does entry data really reach the vulnerable parameter/state? Re-trace independently. |
| Protective mechanisms | Are bounds checks, locks, permission checks, NULL checks actually in effect on this path? |
| Exploit condition realism | Are exploit preconditions satisfiable in the target's actual runtime environment? |
| Compensating controls | Do compiler mitigations (stack canary, ASLR, W^X) or hardware protections (SMMU, MPU, TZASC) eliminate or downgrade the impact? |
| Code misinterpretation | Did Step 2 misunderstand the semantics? (e.g., a variable that appears unvalidated was actually checked upstream) |

Record each check as: PASS (vulnerability real) / FAIL (false positive) / UNCERTAIN (keep but lower confidence)

### Layer 3: Cross-Review

- Each finding reviewed by exactly **one sub-agent**, different from the one that discovered it
- Reviewer receives only: the individual finding context (vuln report entry + vulnerability function source + caller source) — NOT other findings from the same entry, to prevent pattern bias
- Reviewer outputs: CONFIRMED / FALSE_POSITIVE (with reason) / DISPUTED (with reason)
- DISPUTED findings resolved by the main agent (not a sub-agent)

### Parallel Execution

- Each finding assigned to exactly one reviewer sub-agent
- Assignment rule: no sub-agent reviews its own findings
- Each `<uid>_vuls.md` file has only one writer (the main agent), which updates it after receiving review results — no merge conflicts

### Step 3 Output: Updated `<uid>_vuls.md`

For each finding, the main agent fills the `### Review (Step 3)` block:
- `Reviewed` → yes
- `Result` → CONFIRMED / FALSE_POSITIVE
- `Reviewer` → sub-agent identifier
- `Revised Confidence` → only if confidence changed
- `Exclusion Reason` → only if FALSE_POSITIVE

Update per-file `## Summary` table with confirmed/false-positive counts.

### Edge Cases

- **Empty results (no confirmed vulnerabilities)**: Generate `Vul_report.md` with Summary showing all zeros and a note "No confirmed vulnerabilities found." Do NOT omit the report.
- **All entries failed**: Generate `Vul_report.md` with the `## Failed Entries` section listing all entries and their failure reasons. Summary shows 0 confirmed vulnerabilities.
- **Sub-agent failure in Step 3**: Same retry logic as Step 1/2. A failed review is retried once; if still failing, the finding is left as unreviewed and recorded in `## Failed Entries`.

## SKILL.md Entry Point

Orchestration flow (programmatic, linear instructions):

1. **Parse arguments**: `project_dir` from user input (default: current directory)
2. **Read `.siakamignore`**: Load exclusion patterns from `<project_dir>/.siakamignore`
3. **Verify inputs**:
   - Check `<project_dir>/.siakam_out/SII/apis.json` exists → if not, prompt user for path → if still not found, abort
   - Check `<project_dir>/.siakam_out/callgraph.json` exists → set `HAS_CALLGRAPH` flag
   - If `apis.json` has empty `interfaces` array → warn and exit
4. **Initialize**: Create `.siakam_out/SAA/` directory structure. Create `.siakam_out/SAA/tasks.md` with task checklist for all entries across all three steps.
5. **Stage 1**: Read `steps/step1_attack_path.md`. Launch parallel sub-agents (one per entry). Each sub-agent writes `<uid>_attack_path.md`. Update tasks.md as each completes.
6. **Stage 1 failure handling**: After all sub-agents complete, collect failed entries. Retry each failed entry once (new sub-agent). If still failed, mark as FAILED in tasks.md and record in a failures manifest.
7. **Stage 2**: Read `steps/step2_vuln_analysis.md`. For each entry with a successful Stage 1 output, launch a sub-agent (grouped by function if >10 paths per entry). Each sub-agent writes `<uid>_vuls.md` (Step 2 section only). Update tasks.md.
8. **Stage 2 failure handling**: Same retry logic as Stage 1.
9. **Stage 3**: Read `steps/step3_false_positive.md`. Assign each finding to a reviewer sub-agent (different from discoverer). Main agent collects review results and updates `<uid>_vuls.md` files. Update tasks.md.
10. **Stage 3 failure handling**: Retry failed reviews once. If still failing, leave finding unreviewed and record in failures manifest.
11. **Consolidation**: Compile `Vul_report.md` from all CONFIRMED findings across all `<uid>_vuls.md` files.
12. **Cleanup**: Remove `.siakam_out/SAA/.step1_state/` temporary directory. Keep `tasks.md` for audit (or remove if user configures).

Each stage reads its step document at invocation time. Stage transitions pass state via filesystem (previous stage's output files = next stage's input).

### Resume / Checkpoint

If the skill is interrupted and re-invoked:

1. Check if `.siakam_out/SAA/tasks.md` exists
2. If it exists, compare completed tasks (marked `[x]`) against the checklist
3. Verify that output files for completed tasks exist on disk
4. Skip completed tasks; only launch sub-agents for unchecked or failed tasks
5. If tasks.md does not exist, start from Stage 1

## Task Tracking

- Single markdown checklist file: `.siakam_out/SAA/tasks.md`
- Created at initialization, updated atomically after each sub-agent completes
- Survives interruptions for resume capability
- Deleted after successful final consolidation (or kept for audit)
- Format:

```markdown
# Task Tracker

**Started**: <timestamp>
```

## Design Decisions Summary

| Decision | Rationale |
|----------|-----------|
| Three-step pipeline (not single pass) | Separates concerns; avoids context pollution between path-finding, vuln-discovery, and de-duping |
| One sub-agent per entry per step (with grouping in Step 2 for large path counts) | Entries are independent; function-based grouping avoids duplicate analysis |
| Rules/templates embedded in step docs | Self-contained; LLM loads everything for one phase at once |
| File system for inter-phase state | Avoids in-context bloat; files are the contract between phases |
| "Reception = infection" in data flow | Ensures path completeness; pass-through functions don't break trace chains |
| Layer 3 cross-review with different sub-agents | Independent perspective is the strongest defense against confirmation bias |
| Only CONFIRMED vulns in final report | Matches the principle: "Better to miss some than flood the report" |
| English for all prompts | Consistent with original requirements |
| Depth limit 10, infected function limit 50, both configurable | Prevents context explosion; tunable per project |
| Programmatic linear-instruction style for SKILL.md | Reduces LLM ambiguity; ensures consistent execution order |
| Structural markers (`<!-- SECTION -->`) in output files | Enables reliable parsing across stages |
| Composite findings (root cause, not cascading consequences) | Avoids duplicate reporting; clearer for remediation |
| Per-function 8-method checklist with applicability gating | Prevents method omission; reduces noise on inapplicable checks |
| Checkpoint/resume via tasks.md | Survives interruptions without redoing work |
| Sub-agent retry-once-then-skip | Balances reliability with forward progress |
| Function Index in attack_path.md | Enables efficient deduplication and cross-referencing in Steps 2/3 |
| Category × Severity cross-table in final report | Gives security engineers actionable prioritization data |
